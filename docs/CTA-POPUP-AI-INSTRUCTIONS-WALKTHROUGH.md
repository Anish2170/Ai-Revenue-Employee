# CTA Popup and AI Instructions Walkthrough

## Purpose

This walkthrough documents the investigation into missing CTA buttons in generated popups, the exact root cause, and the fixes applied to prevent conversion popups from reaching the widget without a valid Business Action.

It also documents what was found while tracing whether onboarding AI Behavior Instructions are used by both chat responses and popup generation.

## Original Symptom

The popup was visible and the title/body rendered correctly, but no CTA button appeared.

The AI Decision Log showed:

- Popup Generated: Yes
- Validation Passed: Yes
- CTA Type: compare_options
- CTA Action ID: -
- CTA Text: -

That meant the system generated and accepted a popup, but no resolved Business Action reached the widget.

## Flow Traced

The complete popup flow was traced through:

1. Behavior Engine
2. Sales Brain
3. Popup Prompt
4. Gemini Response
5. Popup Validation
6. Business Action Resolution
7. API Response
8. Widget Rendering

The important handoff was:

- The popup pipeline produced validated popup language.
- The validated popup could contain only title/body/tone/popupType, with no `primaryAction`.
- The backend converted the popup into a public artifact.
- The widget rendered buttons only when it received `decision.action` or `decision.secondaryActionConfig`.
- Because no valid action object was attached, the widget correctly rendered no button.

## Root Cause

The root cause was backend validation.

`validatePopupResponse()` allowed conversion-oriented popup types to pass validation even when no valid Business Action was present.

So a popup could be:

- generated by the LLM,
- marked validation-passed,
- logged as having a CTA intent such as `compare_options`,
- delivered to the widget,
- but still contain no resolved `action` object.

This was not a widget rendering bug. The widget was doing what it was coded to do: render a CTA only when a resolved action object exists.

## Hypotheses Checked

| Hypothesis | Result | Evidence |
|---|---:|---|
| Business Actions are disabled, so backend strips CTAs | Partly true | If no enabled executable Business Action exists, the backend could not resolve a CTA. This is now logged clearly. |
| LLM is generating popups without `primaryAction` / `secondaryAction` | Confirmed | Existing validation allowed omitted actions. |
| Popup validator allows CTA-less popups even though type requires actions | Confirmed | This was the main root cause. |
| Backend loses the `actionId` while resolving Business Actions | Rejected | When an enabled action ID exists, backend resolves it into an action object. |
| Widget receives the popup correctly but ignores actions | Rejected | Widget renders `decision.action` and `secondaryActionConfig`. |

## Fixes Implemented

### 1. Required Action Validation

Conversion-oriented popup types now require a valid Business Action.

Strict action requirement applies to:

- `comparison`
- `pricing`
- `booking`
- `lead`
- `support`

If no valid action exists, validation fails with:

```text
missing_business_action
```

This prevents CTA-less conversion popups from reaching the widget.

### 2. Fallback Actions

If the LLM omits an action, the validator now applies a fallback only when an enabled Business Action exists.

Fallback mapping:

| Popup / Intent | Fallback order |
|---|---|
| comparison / compare_options | `pricing`, then `learn_more` |
| pricing / discuss_pricing | `pricing` |
| lead / capture_lead | `book_demo`, then `contact_sales` |
| booking / book_demo | `book_demo`, then `contact_sales` |
| support / offer_support | `contact_support` |
| learn_more | `learn_more` |

No URLs are invented. Destinations still come only from configured Business Actions.

### 3. Strict Validation

Only informational popup types may be shown without CTA actions.

Conversion popups now either:

- resolve a valid Business Action,
- apply an enabled fallback Business Action,
- or are suppressed before widget delivery.

### 4. Better Decision Logging

The AI Decision Log was extended with action-debug fields:

- Expected Action
- Primary Action Returned
- Fallback Applied
- Fallback Used
- Missing Action Reason

Missing action reasons include:

- `LLM omitted action`
- `Business action disabled`
- `Unknown action id`

These fields make it clear why a CTA did or did not appear.

### 5. Business Action Resolution

The public popup response now resolves:

- `primaryAction` into `action`
- `secondaryAction` into `secondaryActionConfig`

The widget receives full Business Action objects and renders buttons from those objects.

## Widget Finding

The widget was not the cause.

The widget button logic is:

- render primary CTA from `decision.action`
- render secondary CTA from `decision.secondaryActionConfig`
- fall back to legacy `decision.cta` only when present

So if the backend sends no resolved action, no button appears.

## AI Behavior Instructions Finding

The onboarding schema already contained fields such as:

- Business Name
- Company Description
- Role
- Tone
- Goal
- Context
- Rules
- Fallback Message
- Website URL

However, the runtime `BusinessInstructions` type and tenant resolver previously exposed only a smaller subset:

- businessName
- tone
- alwaysBookDemo
- avoidDiscounts
- language

This meant several onboarding fields were stored but not reliably injected into chat or popup prompts.

## AI Instruction Fixes

The instruction contract now includes:

- `companyDescription`
- `role`
- `goal`
- `context`
- `rules`
- `fallbackMessage`
- `websiteUrl`

The tenant resolver now maps these fields from `BusinessInstruction` into runtime instructions.

The shared prompt renderer now includes them in `BUSINESS INSTRUCTIONS`, which is used by chat prompts and popup prompts.

## Instruction Usage Table

| Field | Used in Chat | Used in Popups | Influences Output | Evidence |
|---|---:|---:|---:|---|
| Business Name | Yes | Yes | Yes | Rendered in shared instructions and prompt identity |
| Company Description | Yes | Yes | Yes | Added to runtime instructions and shared prompt renderer |
| Role | Yes | Yes | Yes | Rendered as `Assistant role` |
| Tone | Yes | Yes | Yes | Rendered as `Tone`; popup strategy also carries tone |
| Goal | Yes | Yes | Yes | Rendered as `Primary goal` |
| Context | Yes | Yes | Yes | Rendered as `Additional context` |
| Rules | Yes | Yes | Yes | Rendered as `Owner rules` |
| Fallback Message | Yes | Yes | Partial | Rendered in prompts; chat also has separate grounded fallback logic |
| Website URL | Yes | Yes | Limited | Rendered as context; not used to invent CTA destinations |

## Final Conclusion

The missing CTA button happened because the backend allowed a conversion popup to pass validation without a resolved Business Action.

The fix makes the backend responsible for enforcing CTA requirements before delivery. The widget remains simple and safe: it renders only resolved backend-provided Business Actions.
