# Analytics UX Refactor Walkthrough

## Purpose

This refactor separates two different product experiences that previously lived on the same Analytics page:

- Business owner analytics
- Developer AI decision debugging

The goal is product architecture cleanup, not a backend redesign.

Analytics should now answer:

```text
How is my AI Employee performing today?
```

AI Decision Log should now answer:

```text
Why did the AI generate, suppress, validate, display, click, or dismiss a popup?
```

## What Changed

The Analytics page is now business-focused only.

It no longer renders:

- AI Decision Log
- Decision cards
- Behavior details
- Intent details
- Confidence and reason debugging
- Strategy debugging
- Validation details
- LLM debugging
- Decision export controls

Those tools moved into a dedicated AI Decision Log page.

## Main Frontend Files

Changed files:

- `dashboard/src/components/analytics-view.tsx`
- `dashboard/src/app/(dashboard)/layout.tsx`

New file:

- `dashboard/src/app/(dashboard)/ai-decision-log/page.tsx`

No backend APIs, tracking, database models, decision engine logic, popup engine logic, LLM logic, knowledge logic, website actions, or lead capture behavior were changed.

## Analytics Page

Route:

```text
/analytics
```

Component:

```text
AnalyticsView
```

The page now behaves like a business dashboard. It keeps the existing dark SaaS design language and shows real values only where existing APIs expose them.

Displayed sections:

- Top KPI cards
- Filters
- Visitor analytics
- Conversation analytics
- Lead analytics
- Sales funnel
- Top questions
- Website action analytics
- Knowledge analytics
- Website performance
- Recent leads

## KPI Cards

The dashboard includes the requested business KPI surface:

- Today's Visitors
- Unique Visitors
- Returning Visitors
- Today's Conversations
- Messages
- AI Responses
- Leads Captured
- Lead Conversion Rate
- Popup CTR
- Website Action CTR
- Email Capture Rate
- Average Response Time
- Average Conversation Length
- Escalated Conversations
- Resolved Without Human

Important rule:

If a metric is not currently available from the existing analytics API, the card displays a clear placeholder:

- `Unavailable`
- `Coming Soon`
- `No data yet`

It does not invent numbers.

## Existing Data Reused

The page still uses the existing API client methods:

```text
api.getAnalyticsSummary()
api.getAnalyticsChart()
```

Real existing values currently reused include:

- today's visitors
- today's conversations
- messages
- AI responses
- popup displayed count
- popup clicked count
- popup CTR
- daily visitors chart
- daily chats chart
- top pages
- device breakdown
- top popup types
- website performance

## Empty States

The dashboard avoids broken or fake UI states.

Charts, metric groups, and tables now use explicit empty states such as:

- `No data yet`
- `Unavailable`
- `Coming Soon`

The page should not show:

- `undefined`
- `null`
- `NaN`

## Export Behavior

The decision-log export behavior was preserved.

The UI still calls:

```text
api.getAiDecisionLog({
  startDate,
  endDate,
  limit: 5000,
  export: true
})
```

The browser still downloads a readable `.txt` export generated by the existing frontend export builder.
